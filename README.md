1分版のプロジェクト紹介
「このプロジェクトは、企業内ナレッジベース向けの Role-Aware RAG システムです。
通常のRAGと違って、単に回答できるだけではなく、ユーザーの権限に応じて参照できる文書を制御する点が特徴です。
文書としては主に Excel や PDF を取り込み、解析して chunk に分割したうえで、ベクトルDBに保存します。各 chunk には allowed_roles のような権限情報や、元のファイル名、sheet名、cell range などのメタデータを持たせています。
ユーザーが質問すると、まず現在のロールで参照可能な chunk に絞り込み、その後 vector、bm25、hybrid、hybrid_rerank などの検索方式で関連情報を取得します。最後に、その検索結果だけを根拠としてLLMに回答させることで、権限制御と回答の根拠性を両立しています。
技術スタックは、バックエンドが FastAPI、フロントエンドが React + Vite、ベクトルストアが Qdrant、DB が PostgreSQL です。」
30秒版の短縮紹介
「企業内文書向けの権限制御付きRAGシステムです。ExcelやPDFを解析して chunk 化し、各 chunk にロール情報を付与して Qdrant に保存します。質問時には、ユーザー権限で参照可能な chunk のみを検索し、その結果を根拠としてLLMに回答させます。通常のRAGよりも、検索層でアクセス制御している点が大きな特徴です。」
このプロジェクトで特に話すべきポイント
一番の特徴は、権限制御をUIではなく検索層で行っていることです。
単純なチャットボットではなく、企業内ナレッジ検索の実運用を意識した設計です。
Excel仕様書のような複雑な文書に対応するため、文書解析と chunk 設計に工夫がある点も強みです。
検索方式を vector / bm25 / hybrid / hybrid_rerank で切り替えられ、意味検索とキーワード検索の両方を使い分けられる点も説明しやすいです。
幻覚対策として、検索結果を reference として与え、その範囲内でのみ答えるよう制約している点も話せます。
面接で聞かれやすい質問と回答例
このプロジェクトの一番の特徴は何ですか？
「一番の特徴は、RAG にロールベースのアクセス制御を組み合わせている点です。通常のRAGは関連文書を探して回答することが主目的ですが、このプロジェクトでは、ユーザーの権限に応じて検索対象そのものを制限しています。つまり、回答生成の前段階で越権アクセスを防いでいます。」

なぜ権限制御を検索層でやる必要があるのですか？
「UIだけで制御すると、内部的には機密文書が検索対象になってしまう可能性があります。RAGでは、検索された文書がそのまま回答の根拠になるので、検索段階で権限を制御しないと、見せてはいけない情報をモデルが参照してしまうリスクがあります。そのため、検索層で allowed_roles によるフィルタを入れる設計が重要です。」

chunk とは何ですか？ なぜ必要ですか？
「chunk は文書を分割した最小の検索単位です。文書全体をそのまま検索や生成に使うと、文脈が大きすぎて精度や効率が悪くなります。そのため、適切な長さに分割した chunk 単位でベクトル化・検索を行います。このプロジェクトでは、各 chunk に権限や出典情報も持たせています。」

このプロジェクトでは、どうやって chunk を作っていますか？
「まず Excel や PDF を解析して、検索しやすい統一テキストに変換します。その後、SentenceSplitter で一定サイズごとに分割し、さらに sheet_name や cell_range などの構造情報を各 chunk に紐付けています。単純に文字数だけで切るのではなく、元文書の構造も追跡できるようにしているのがポイントです。」

分かち書きや NLP の分詞ではなく、なぜ chunk 分割なのですか？
「分詞は BM25 のようなキーワード検索には有効ですが、RAG の検索単位としては細かすぎます。モデルに渡すには、単語の列ではなく、意味がまとまった短い文章単位のほうが適しています。そのため、このプロジェクトでは、検索単位としては chunk を使い、BM25 側では別途 Sudachi による分詞を使っています。」

vector、bm25、hybrid、hybrid_rerank の違いは何ですか？
「vector は意味ベースの検索、bm25 はキーワードベースの検索です。hybrid はその両方を組み合わせて、意味検索の強みとキーワード検索の強みを両立します。hybrid_rerank はさらに reranker を使って候補を再順位付けするので、精度が高くなりますが、その分コストやレイテンシは上がります。」

このプロジェクトで使っているモデルは何ですか？
「デフォルト構成では、生成用に qwen3.5:4b、embedding 用に bge-m3、rerank 用に BAAI/bge-reranker-v2-m3 を使っています。全体としては OpenAI-compatible な形で接続しているので、将来的な差し替えもしやすい構成です。」

RAG はどこで普通のNLPと違いますか？
「普通のNLPは、テキストそのものを解析して分類・抽出・分詞などを行うことが中心です。一方RAGは、まず外部知識を検索して、その結果を根拠として生成を行う仕組みです。このプロジェクトでも、NLP 的な要素は分詞や関連度計算にありますが、中心はあくまで検索と生成のパイプラインです。」

このプロジェクトでは、どうやって幻覚を抑えていますか？
「モデル任せではなく、コード側で制約を入れています。具体的には、まず検索結果がない場合は回答を作らずに ‘該当情報なし’ を返します。検索結果がある場合も、その reference 情報だけを使って回答するよう prompt で制約し、回答には citation を含めるようにしています。つまり、幻覚をゼロにするというより、回答の根拠を検索結果に寄せることでリスクを下げています。」

Qdrant と PostgreSQL を両方使っている理由は何ですか？
「役割が異なります。PostgreSQL はユーザーや認証情報などの業務データ管理用で、Qdrant は chunk のベクトルとメタデータを持つRAG用の知識ストアです。つまり、業務DBと検索用ベクトルDBを分けています。」

このプロジェクトで技術的に難しい点はどこですか？
「LLM接続そのものより、文書解析と検索精度の設計が難しいと考えています。特に複雑なExcel仕様書は、単純テキスト化すると構造が崩れやすいので、sheetやcell rangeを保ちながら chunk 化して、検索精度と出典追跡性を両立させるところがポイントです。」

もし改善するとしたら、どこを改善しますか？
「改善ポイントとしては、回答の根拠チェックをさらに強くすることです。例えば、citation が付いているだけでなく、回答文の各主張が本当に source に支えられているかを検証する仕組みを追加したいです。また、chunk サイズや rerank 条件のチューニング、評価指標の継続的な改善も進めたいです。」

最後に面接で使いやすい締めの一言
「このプロジェクトは、単なるLLM活用ではなく、企業文書検索として必要な権限制御、文書構造保持、検索精度、根拠提示まで含めて設計されている点が面白いと感じています。」

Linux経験はどの程度ですか
Linux環境で動作するPythonアプリケーションの改修と、Dockerコンテナの基本的な運用経験があります。手順書に基づく停止、起動、リリース、動作確認や、docker execを使ったコンテナ内の確認を行いました。  
一方で、Linuxサーバー自体の設計・構築や高度な障害対応については、経験が限定的です。

Docker経験はどの程度ですか
コンテナの停止・起動、状態確認、docker execによるコンテナ内部の確認など、基本的な操作経験があります。Docker基盤の設計や大規模なコンテナオーケストレーションについては、まだ実務経験がありません。

AWSではEC2とECSのどちらを使いましたか
AWS基盤の構成は担当範囲外だったため、EC2かECSかについては正確に把握していません。私自身はLinux・Docker環境上で、Pythonアプリケーションの改修、テスト、リリースを担当しました。

実際に複雑な障害対応がない場合：
大規模な本番障害の原因分析や恒久対策を主担当として実施した経験はありません。これまでの業務では、問い合わせや問題発生時に、発生条件、ログ、処理結果を確認し、アプリケーション側の問題か、データや環境側の問題かを切り分ける対応を行いました。

BGPとは何ですか
BGPは、異なる自律システム間で、どのIPネットワークへどの経路で到達できるかというルーティング情報を交換するためのプロトコルだと理解しています。ASNは、その自律システムを識別する番号です。

VPCとOCXの違いは何ですか
VPCは基本的にAWSなどのクラウド内部に作成する仮想ネットワークです。一方、OCXは企業拠点、データセンター、複数のパブリッククラウドをまたいで接続するネットワークサービスだと理解しています。

なぜインターネットVPNでは不十分なのですか
小規模な用途ではインターネットVPNでも対応できますが、通信量が多い場合や、安定した帯域・遅延、閉域接続、複数クラウドの一元管理が必要な場合には、OCXのような専用サービスに価値があると考えています。
